## Part 1 — Answers to the open questions in implementation-plan.md (pre-revision)

2. **OSHA 301 employee DOB / hire-date / gender into existing `type_data` JSON** — approved. Document in `docs/compliance-notes.md` that these keys are load-bearing for OSHA 301 PDF generation so future developers don't treat `type_data` as freely mutable.

3. **Retention satisfied by absence of deletion endpoints** — approved. Document explicitly in `docs/compliance-notes.md` as a deliberate design choice. If any future feature would add a delete capability, it must hard-block deletion of records inside their regulatory retention window.

4. **"Draft" state** — confirmed descoped. No widening of `incidents.status` CHECK. If a server-side draft store is ever required, do it as a separate `incident_drafts` table.

5. **GDPR Article 17 erasure** — descoped. Document in `docs/compliance-notes.md` that:
   - GDPR Art 17(3)(b) carves out processing required by legal obligation.
   - OSHA / RIDDOR / WHS retention periods constitute such legal obligation.
   - Eventual implementation will be: `erasure_requests` table, admin workflow, retention-period block, audit entries for the erasure itself, and in-place PII nullification (UPDATE name etc. to NULL, keep the incident facts).

6. **Encryption at rest** — deployment requirement, no application code. Document in `docs/compliance-notes.md` that the data directory (SQLite file + attachments) must be on an encrypted filesystem in production. SQLCipher is an alternative if the platform cannot guarantee filesystem encryption.

7. **WI-09 (Generic Incident PDF)** — keep in scope. It is the universal fallback for non-regulatory reporting and the only PDF available for incidents outside the supported jurisdictions.

## Part 2 — Three new additive work items

These address real product weaknesses surfaced in product review. Each is strictly additive. **The no-structural-changes directive applies in full**: do not refactor existing tables, remove existing columns, or change existing API contracts.

### WI-A — Multi-person incident support

**Problem.** Today `incidents.type_data.injured_person` holds a single injured person as JSON. Real incidents sometimes involve multiple affected people; the current model forces them into separate linked incidents, fragmenting investigation and reporting.

**Approach.** Introduce two new tables that *supplement, not replace*, the existing JSON. The legacy single-person JSON stays in place untouched. A one-time backfill copies each existing JSON sub-record into the new tables as the primary affected person. New incidents may use either path.

**Schema (new migration):**

`affected_persons`:
- `id` INTEGER PK
- `incident_id` FK to `incidents`, NOT NULL
- `org_id` NOT NULL (denormalized for tenant scoping)
- `name`, `dob` (ISO date), `gender`, `address`, `phone`, `email`
- `job_title`, `employment_status` (`employee | contractor | labour_hire | volunteer | visitor | member_of_public | self_employed`)
- `employer_name`, `date_hired` (ISO date), `experience_years`, `hours_into_shift`
- `is_privacy_case` (for OSHA 1904.29(b)(7))
- `is_primary` (exactly one per incident; this is the row that mirrors the legacy JSON sub-record)
- `deleted_at` (soft delete), `created_at`, `created_by`, `updated_at`, `updated_by`
- Index on `(incident_id, deleted_at)`, index on `(org_id, incident_id)`

`injuries`:
- `id` INTEGER PK
- `affected_person_id` FK, NOT NULL
- `org_id` NOT NULL
- `body_part`, `injury_type`, `mechanism`, `object_substance`
- `treatment`, `physician_name`, `physician_phone`, `physician_facility`
- `er_treated`, `hospitalized`, `hospitalization_date`
- `days_away`, `days_restricted`, `date_of_death` (ISO date)
- `narrative`
- `deleted_at`, `created_at`, `created_by`, `updated_at`, `updated_by`
- Index on `(affected_person_id, deleted_at)`

**Backfill.** For every existing incident with `type_data.injured_person` populated, insert one `affected_persons` row (`is_primary = 1`) and one `injuries` row reflecting the JSON contents. For incidents where `type_data` is NULL or has no `injured_person` sub-record, skip — don't create empty rows.

**Existing routes — dual-write:**
- `POST /incidents` and `PATCH /incidents/:id` continue to accept the existing `type_data.injured_person` shape (backward compatible).
- When a request arrives with the legacy single-person shape, the route writes BOTH the `type_data` JSON AND a corresponding `affected_persons` row (`is_primary = 1`) plus one `injuries` row.
- When a request arrives with a new multi-person shape (an `affected_persons` array), the route writes ONLY the new tables and populates `type_data.injured_person` with a copy of the primary person for backward compatibility with any code still reading the JSON.

**New routes (additive):**
- `GET /incidents/:id/affected-persons` — returns list with nested injuries
- `POST /incidents/:id/affected-persons` — add a new affected person plus their injuries in one call
- `PATCH /incidents/:id/affected-persons/:apId`
- `DELETE /incidents/:id/affected-persons/:apId` (soft delete)
- `POST /incidents/:id/affected-persons/:apId/injuries`
- `PATCH /incidents/:id/affected-persons/:apId/injuries/:injuryId`
- `DELETE /incidents/:id/affected-persons/:apId/injuries/:injuryId` (soft delete)

All routes scoped by `org_id`. All mutations write to `activity_log`.

**Client work:**
- `client/src/api/incidents.js` — add the new API functions.
- `client/src/pages/incidents/IncidentDetail.jsx` — show N affected persons each with N injuries; current single-person UI continues to render for legacy data where only `is_primary` exists.
- **Wizard work for capturing multi-person incidents is OUT OF SCOPE for this WI.** The new tables exist and are reachable via API and incident detail UI; wizard expansion is a separate future WI.

**Acceptance:**
- Migration applies cleanly and backfills existing data.
- Existing incidents still render correctly in the detail view (one primary person from backfill).
- A new incident can be created via API with multiple affected persons.
- Legacy `POST /incidents` requests with `type_data.injured_person` still work and create both JSON and new-table rows.
- All existing tests pass; new tests cover dual-write and the new endpoints.

### WI-B — Recordability override approval workflow

**Problem.** Today any user with permission to edit an incident can flip `osha_recordable` / `riddor_reportable`. There is no separation of duties between the person who proposes a change and the person who approves it. This is a real internal-control gap.

**Approach.** Introduce a small override-request flow that wraps the existing boolean. The existing columns stay as the source of truth; they only change when an override request is approved.

**Schema (new migration):**

`classification_override_requests`:
- `id` INTEGER PK
- `incident_id` FK, NOT NULL
- `org_id` NOT NULL
- `jurisdiction` TEXT NOT NULL CHECK (jurisdiction IN (`'US-OSHA'`,`'UK-RIDDOR'`,`'AU-NSW'`))
- `field` TEXT NOT NULL (e.g. `'osha_recordable'`, `'riddor_reportable'`)
- `current_value` INTEGER (the boolean value at time of request)
- `proposed_value` INTEGER NOT NULL
- `reason` TEXT NOT NULL (required justification)
- `status` TEXT NOT NULL CHECK (status IN (`'pending'`,`'approved'`,`'rejected'`,`'withdrawn'`)) DEFAULT `'pending'`
- `requested_by` INTEGER FK to users, NOT NULL
- `requested_at` TEXT NOT NULL DEFAULT CURRENT_TIMESTAMP
- `decided_by` INTEGER FK to users (NULL until decided)
- `decided_at` TEXT
- `decision_note` TEXT
- Indexes on `(org_id, status)`, `(incident_id, status)`

**Rules:**
- `requested_by != decided_by` (self-approval forbidden; enforce in route, also as a DB CHECK if SQLite version supports it, otherwise as a trigger).
- Only users with role in `{supervisor, ehs_officer, ehs_manager, admin}` may decide.
- Approving a request flips the underlying boolean on `incidents` AND writes `activity_log` entries for both the override approval and the boolean change.
- Rejecting or withdrawing leaves the boolean untouched.
- An incident may not have more than one pending request per `(field)` at a time.

**New routes:**
- `POST /incidents/:id/override-requests` — create.
- `GET /incidents/:id/override-requests` — list for this incident.
- `GET /override-requests?status=pending` — global pending queue for approvers, org-scoped.
- `POST /override-requests/:reqId/approve`.
- `POST /override-requests/:reqId/reject`.
- `POST /override-requests/:reqId/withdraw` — only the requester may withdraw.

**Existing direct-edit path:** keep working for now, but emit a `console.warn` server-side when `osha_recordable` or `riddor_reportable` is PATCHed directly without going through an override request. This lets us measure direct-edit usage before deciding whether to forbid it in a later WI.

**Client work:**
- `client/src/api/incidents.js` — add the override request functions.
- `client/src/components/incidents/RecordabilityVerifyCard.jsx` — replace the direct "Override" button with "Request override" which opens a modal capturing reason; show pending request state inline.
- New page or panel: pending approvals queue for users with approver roles. Reuse existing list/card styling per CLAUDE.md tokens.

**Acceptance:**
- An incident's recordable boolean cannot be changed via the new flow by the requester alone.
- The full request → decision lifecycle is visible in `activity_log`.
- A pending request blocks a second pending request on the same field.
- The UI surfaces pending requests on the incident detail and via a global queue.

### WI-C — Activity log integrity (chain of custody)

**Problem.** `activity_log` is append-only by convention but not by construction. Anyone with database access could insert, update, or delete entries without detection.

**Approach.** Add a hash chain and append-only triggers. Existing entries are backfilled into the chain. Existing log writers are updated to include the chain values and missing per-entry metadata uniformly. No existing columns are removed; only new columns are added.

**Schema (new migration):**

`ALTER TABLE activity_log ADD COLUMN`:
- `prev_hash` TEXT
- `entry_hash` TEXT
- `ip_address` TEXT (already exists on `regulatory_certifications`; bring it to `activity_log`)
- `user_agent` TEXT
- `field_diffs` TEXT (JSON; structured diff for UPDATE operations)

**Backfill.** For existing rows ordered by `(created_at, id)`, compute the chain:
- `prev_hash` for row N = `entry_hash` of row N-1 (NULL for the first row in an org).
- `entry_hash` for row N = SHA-256 of `(prev_hash || canonical_json(row))` where `canonical_json` normalizes whitespace and key order.
- Chain is per `org_id` (each tenant has its own chain head).

**Append-only enforcement:**
- `AFTER UPDATE OF prev_hash, entry_hash, ip_address, user_agent, field_diffs, action, entity_type, entity_id, actor_id, org_id, created_at, metadata ON activity_log` → `RAISE(ABORT, 'activity_log is append-only')`.
- `AFTER DELETE ON activity_log` → `RAISE(ABORT, 'activity_log entries cannot be deleted')`.
- Triggers may be bypassed only by explicit schema migrations that disable them temporarily.

**Service changes** — `server/services/activity_log.js` (the centralized module; update in place):
- `writeActivity({ orgId, actorId, action, entityType, entityId, ip, userAgent, fieldDiffs, metadata })` is the single writer.
- It reads the latest `entry_hash` for the org, computes the new `entry_hash`, inserts the row.
- Existing call sites across routes update to pass `req.ip`, `req.get('user-agent')`, and a `fieldDiffs` structure when relevant.
- Helper: `diffFields(before, after, watchedFields)` returns the structured diff to pass as `fieldDiffs`.

**Verification:**
- New endpoint `GET /reports/audit-log/verify` returns chain integrity status per org: `{ ok: true, count }` or `{ ok: false, brokenAt }`.
- This is an internal-admin endpoint; restrict to `admin` role.

**Acceptance:**
- Migration applies, backfills all existing rows into a valid chain.
- Verify endpoint returns `ok: true` immediately after migration.
- A test that manually UPDATEs an `activity_log` row is blocked by the trigger.
- A test that calls `writeActivity` multiple times produces a chain where each entry's `prev_hash` equals the previous entry's `entry_hash`.
- All existing routes that write to `activity_log` continue to function, now also populating `ip_address`, `user_agent`, and `field_diffs`.

> **Note on the no-structural-changes directive.** WI-C is an explicit owner-authorized override of the "no column additions to existing tables" position. `ALTER TABLE activity_log ADD COLUMN` is permitted here; the directive otherwise still stands for all other tables.
